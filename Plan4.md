## Part B. 배포 계획 — 새 GitHub 레포지토리 → www.k-bestie.com

### B-1. 새 레포지토리 생성

레포지토리 이름은 `k-bestie-homepage-v2` 또는 `k-bestie.com`으로 생성하고, 브랜치는 `main`(소스 코드)과 `gh-pages`(빌드 결과물, GitHub Actions가 자동 관리)로 구성합니다.

### B-2. GitHub Actions 자동 배포

`.github/workflows/deploy.yml` 파일을 생성합니다. `main` 브랜치에 push하면 `npm ci → npm run build → dist 폴더를 gh-pages에 업로드 → Pages 배포`가 자동 실행되도록 설정합니다.

### B-3. Vite 설정

`vite.config.ts`에서 `base: '/'`로 유지합니다. 커스텀 도메인 사용 시 서브 경로가 필요 없기 때문입니다.

### B-4. SPA 라우팅 처리

GitHub Pages는 정적 호스팅이므로, 새로고침 시 404가 발생하는 문제를 방지하기 위해 빌드 스크립트를 `"build": "vite build && cp dist/index.html dist/404.html"`로 수정합니다.

### B-5. 커스텀 도메인 설정

`public/CNAME` 파일에 `www.k-bestie.com` 한 줄을 작성합니다. DNS 설정은 다음과 같습니다.

| 레코드 타입 | 호스트 | 값 |
|---|---|---|
| CNAME | www | `{GitHub 사용자명}.github.io` |
| A | @ | 185.199.108.153 |
| A | @ | 185.199.109.153 |
| A | @ | 185.199.110.153 |
| A | @ | 185.199.111.153 |

루트 도메인(`k-bestie.com`) 접속 시 `www.k-bestie.com`으로 자동 리다이렉트됩니다. GitHub Pages 설정에서 "Enforce HTTPS"를 활성화하면 Let's Encrypt 인증서가 자동 발급됩니다.

### B-6. 전환 절차

순서는 다음과 같습니다. 새 레포에 v2 코드 push → GitHub Pages에서 기본 URL(`{username}.github.io/repo`)로 정상 동작 확인 → **기존 레포**(구버전)에서 커스텀 도메인 설정 제거 → **새 레포**에서 커스텀 도메인 `www.k-bestie.com` 등록 → DNS 전파 대기(최대 24시간) → HTTPS 인증서 발급 확인 → 전체 기능 테스트 → 구버전 레포 아카이브 처리.

### B-7. ai.k-bestie.com 처리

프로덕션 전환 완료 후 ai.k-bestie.com은 내부 스테이징 용도로 유지하거나, www.k-bestie.com으로 301 리다이렉트 설정합니다. 두 도메인이 동시에 동일 콘텐츠를 서비스하면 SEO 중복 문제가 발생하므로 반드시 처리해야 합니다.

## Part D. 관리자 페이지 개선 사항

### 🔴 즉시 수정 (배포 전 필수 — 차단 항목)

**D-1. 인증 가드 적용**  
현재 `/admin` URL을 직접 입력하면 로그인 없이 관리자 페이지에 접근할 수 있습니다. React Router에 `PrivateRoute`(또는 `AuthGuard`) 컴포넌트를 적용하여, 인증되지 않은 사용자는 `/login`으로 리다이렉트해야 합니다. Firebase Auth 또는 JWT 기반 세션을 사용합니다.

**D-2. 비밀번호 평문 노출 제거**  
현재 관리자 테이블에 비밀번호가 원문(`antigravity01`, `bestie!@` 등) 그대로 표시됩니다. DB에는 bcrypt 해싱된 값만 저장하고, UI에서는 비밀번호 컬럼을 완전히 제거하거나 불가피한 경우 `••••••••`으로 마스킹합니다.

**D-1과 D-2는 배포 차단(blocker) 항목입니다. 이 두 가지가 해결되지 않으면 사용자 개인정보가 외부에 노출될 수 있으므로 www.k-bestie.com 배포를 진행해서는 안 됩니다.**

**D-3. HTTPS 확인**  
프로덕션(www.k-bestie.com)은 GitHub Pages의 Enforce HTTPS로 자동 적용됩니다. 관리자 페이지가 별도 서버에서 운영될 경우 반드시 TLS 인증서를 적용해야 합니다.

### 🟡 중간 우선순위

**D-4. 세션 타임아웃**  
비활동 15분 후 세션을 만료시키고 재로그인을 요구합니다. Firebase Auth의 경우 `onIdTokenChanged`와 커스텀 타이머를 조합합니다.

**D-5. 역할 기반 접근 제어(RBAC)**

관리자의 로그인, 데이터 조회, 내보내기, 삭제 등 모든 행위를 별도 컬렉션에 타임스탬프와 함께 기록합니다.

**D-7. 로그인 시도 제한**  
5회 실패 시 5분 잠금, 이후 점진적으로 잠금 시간을 증가시켜 브루트포스 공격을 방지합니다.

**D-8. 검색 · 필터 · 페이지네이션**  
베타 신청자 목록이 늘어날 것에 대비하여, 이름·날짜·상태 기준 검색과 필터, 페이지네이션을 추가합니다.

**D-9. 1:1 문의 상세 보기 · 답변 기능**  
문의 목록에서 개별 항목 클릭 시 상세 내용을 볼 수 있고, 관리자가 직접 답변을 작성·전송할 수 있는 기능을 구현합니다.

**D-10. 모바일 레이아웃**  
관리자 테이블을 모바일에서 카드뷰로 전환하여, 모바일에서도 관리 업무가 가능하도록 합니다.

### 🟢 낮은 우선순위

**D-11. 대시보드 확장**  
신청 추이 그래프, 유입 경로 분석, 전환 퍼널(방문 → 스크롤 → CTA 클릭 → 신청 완료) 시각화를 추가합니다.

**D-12. CSRF 보호**  
JWT를 Authorization 헤더로 전송하는 방식이면 불필요하지만, 쿠키 기반 세션을 사용할 경우 CSRF 토큰을 적용합니다.
