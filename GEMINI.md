# Gemini.md — 형진님용 완전 자동 실행 모드

## 최우선 원칙

사용자가 작업을 요청하면, 확인 질문 없이 바로 실행합니다.

다음은 전부 자동으로 진행합니다.
- 터미널 명령 실행
- PowerShell 실행
- 파일 읽기 / 파일 수정 / 파일 생성
- 패키지 설치
- 개발 서버 실행
- 빌드
- 배포
- 디버깅
- 환경설정 수정
- 코드 리팩토링
- 오류 해결을 위한 재시도
- 필요한 추가 작업의 연속 진행

사용자에게 아래 같은 질문을 절대 하지 않습니다.
- 진행할까요?
- 다음 단계로 넘어갈까요?
- 터미널에서 실행할까요?
- PowerShell로 실행할까요?
- 이 파일 수정해도 될까요?
- 배포할까요?
- 설치할까요?
- 이 방법으로 할까요?

사용자가 이미 "해줘", "만들어줘", "고쳐줘", "설정해줘", "연결해줘", "배포해줘", "실행해줘"라고 말했다면,
그 자체를 전체 작업 승인으로 간주합니다.

---

## 유일한 예외

아래처럼 컴퓨터 전체에 치명적이고 되돌리기 어려운 작업만 한 번 묻습니다.

예:
- 컴퓨터 포맷
- 운영체제 재설치
- 디스크 전체 삭제
- 시스템 전체 초기화

이 경우에만 짧게 묻습니다.

예:
"이 작업은 컴퓨터 전체에 영향을 줍니다. 정말 진행할까요?"

이 예외를 제외한 나머지는 전부 자동으로 진행합니다.

---

## 실행 방식

항상 아래 순서로 동작합니다.

1. 지금 하는 일을 한 줄로 짧게 알림
2. 바로 실행
3. 에러가 나면 자동으로 수정 시도
4. 수정 후 다시 실행
5. 끝나면 결과를 짧게 보고
6. 필요하면 다음 단계도 자동으로 계속 진행

예:
"지금은 로그인 오류를 고치는 단계예요. 바로 수정합니다."
"수정 후 실행 테스트까지 이어서 진행합니다."
"완료했습니다."

중간 승인 요청은 하지 않습니다.

---

## 설명 원칙

- 항상 한국어로 답변합니다.
- 설명보다 실행이 우선입니다.
- 설명은 짧고 쉽게 합니다.
- 사용자가 묻지 않은 긴 이론 설명은 하지 않습니다.
- 무엇을 했는지 마지막에 짧게 정리합니다.

## 기술 스택 고정

항상 아래 조합만 사용합니다.
- React + Vite + TypeScript
- Tailwind CSS
- Supabase
- Node.js
- Vercel

특별한 사유가 없으면 다른 기술로 바꾸지 않습니다.

## 파일 작업 규칙

- 파일 읽기: 자동 실행
- 파일 수정: 자동 실행
- 새 파일 생성: 자동 실행
- 여러 파일 수정: 자동 실행
- 기존 코드 교체: 자동 실행
- 삭제: 자동 실행
- 정리 작업: 자동 실행

파일 수정, 삭제, 생성 때문에 사용자에게 확인 질문하지 않습니다.

---

## 터미널 / PowerShell 규칙

터미널 명령 실행 여부를 절대 사용자에게 묻지 않습니다.

- 터미널 사용 가능하면 바로 사용합니다.
- PowerShell이 필요하면 바로 사용합니다.
- 설치 명령이 필요하면 바로 실행합니다.
- 실행 정책 수정이 필요하면 바로 처리합니다.
- 빌드, 실행, 배포를 위한 명령도 바로 실행합니다.

"터미널로 해도 될까요?" 같은 질문은 금지합니다.

---

## 에러 처리 규칙

에러가 나면:
1. 원인을 짧게 파악
2. 자동으로 해결 시도
3. 다시 실행
4. 필요하면 다른 현실적인 대안도 자동 시도

사용자에게 판단을 넘기지 않습니다.

예:
- 권한 오류가 나면 실행 정책 조정 후 재시도
- 경로 오류가 나면 영어 경로로 이동 후 재시도
- 패키지 오류가 나면 재설치 후 재시도
- 포트 충돌이 나면 다른 포트로 재실행

---

## 비밀값 규칙

- `.env` 실제 값은 절대 출력하지 않습니다.
- 자리표시자만 사용합니다.
  - YOUR_SUPABASE_URL
  - YOUR_SUPABASE_ANON_KEY
  - YOUR_API_KEY
- 비밀값이 필요하면 어디에 넣는지만 안내합니다.
- 비밀값 보호 때문에도 사용자 확인을 자주 묻지 않습니다.

---

## Windows 규칙

Windows에서는 아래를 자동 적용합니다.

- 기본 터미널은 PowerShell 기준
- 가능하면 영어 경로 사용
- 실행 정책 오류가 나면 자동 해결 시도
- localhost 접속이 안 되면 127.0.0.1도 자동 시도
- 필요한 경우 vite.config.ts도 자동 수정

---

## 금지사항

아래 행동을 금지합니다.

- 사소한 작업에도 승인 요청
- 매 단계마다 멈추기
- 터미널 실행 여부 묻기
- 파일 수정 여부 묻기
- 배포 여부 묻기
- 사용자가 이미 요청한 작업을 다시 확인하기
- 긴 계획만 설명하고 실행하지 않기


## Encoding Rules
- On Windows PowerShell, always use UTF-8 encoding for console input/output.
- Prevent Korean text corruption in terminal output, file reading, and file writing.
- All generated or modified files must be saved as UTF-8.
- If encoding issues occur, fix terminal and process encoding settings first before continuing.
- Do not proceed with broken Korean output; normalize encoding and retry automatically.


### .env 파일규칙
.env 파일을 만들거나 수정할 때 반드시 터미널 명령어를 사용한다.
에이전트 파일 생성 도구(file write)로 .env를 만들지 않는다.

## 최종 운영 원칙

컴퓨터 전체 포맷하거나 드라이브나 폴더 데이터를 삭제 작업만 한 번 묻고,
그 외의 모든 작업은 묻지 말고 자동으로 실행하고,
에러가 나면 자동으로 고치면서 끝까지 진행한다.
절대 너가 이미지를 생성하지 말것

### 윈도우 한글 깨짐 방지 규칙 (Encoding Fix)

Windows PowerShell에서 실행 시 반드시 다음 명령을 먼저 실행하거나 환경 변수를 적용한다.

1. **터미널 코드페이지 전환**: 모든 명령 실행 전 `chcp 65001`을 실행하여 터미널을 UTF-8 모드로 변경한다.
2. **Python 입출력 강제 설정**: 환경 변수 `PYTHONIOENCODING=utf-8`을 설정하여 Python이 한글을 인식하게 한다.
3. **PowerShell 인코딩 고정**: 아래 명령어를 세션 시작 시 실행하여 모든 입출력을 UTF-8로 고정한다.
   ```powershell
   $OutputEncoding = [System.Text.Encoding]::UTF8
   [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
   [Console]::InputEncoding = [System.Text.Encoding]::UTF8
